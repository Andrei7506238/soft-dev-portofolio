Requirements and objectives

  • Creating a dynamic system that streamlines menu organization and efficiently handles restaurant orders.
  • Developing a modular and flexible architecture that allows for easy functionality expansion through extensions, empowering business owners to adjust the sistem acording to their needs.
  • Crafting a modern, cross-platform application that is user-friendly and customizable.
  • Implementing an intuitive, adaptable system suitable for a variety of usage scenarios.

Features

Plugin System

Extend the system's functionality through plugins, allowing for easy customization and adaptation. The system can load plugins at runtime, making it highly flexible and extensible. Plugin components can interoperate with the main application and other plugins through well-defined interfaces, ensuring smooth interaction.

Cross-Platform Application

The client application is built using Flutter, a cross-platform framework that allows for the development of applications for Android, iOS, and the web. This ensures a consistent user experience across different platforms, reducing development time and costs.

Management System for Menu and Orders

The system provides tools for managing the menu and orders. The menu can be easily organized following a familiar structure. Orders with different menu versions can be handled simultaneously, with the possibility of editing the menu at any time.

Client caching for old menu versions

For orders already made, the menu does not change and this feature reduces the calls to the server semnificantly.

Built-in image management

The server has a builtin support for image storage, reducing the dependency from other services.

Client caching for images

The client caches the images, reducing the number of requests to the server and improving the user experience.

Self-hosted

The server can be self hosted, removing the need for internet access and allowing much more customization.

Clean architecture

The server is built following the clean architecture principles, ensuring a high level of maintainability and scalability.

Customizable orders page

The user has the ability to hide details about the orders, reducing the clutter on the screen.

Statistics page

The manager has access to a page showing a multitude of graphics and statistics about the orders and the menu and see info about finished orders.

Extensible authentication and permission system

The system has a built-in authentication and permission system, allowing for easy customization and extension via plugins.

Project Organization

ASP.NET CORE SERVER

The server is structured following best practices regarding clean architecture, with a clear separation between controllers, services and repositories. Dependencies between components are based on interfaces, ensuring a high level of abstraction and flexibility. The system is designed to be easily extendable, with the ability to add new features through plugins.

The solution is elegantly divided into four key projects:
► Domain, which acts as the backbone, housing interfaces for data repositories and services, entities, and the event and transaction components. This project contains the interfaces that are implemented in the other projects, ensuring a clear separation of concerns. Any developed plugin will have to depend on and implement interfaces from this project, witout needing to depend on the Implementation project;
► Implementation, where the magic happens with concrete implementations, database management, controllers, authentication, real-time communication, and client notification;
► TestImplementation, ensuring reliability through unit tests for verifying functionality;
► Main the powerhouse entry point responsible for loading plugins and kickstarting the system. This modular structure enhances code clarity and manageability while boosting flexibility, making the system robust, scalable, and easily extendable.

FLUTTER CLIENT

The client application is organized into three packages:
► Domain, containing interfaces for entities, data repositories, and a messaging service;
► Implementation, housing implementations for entities, services, data repositories, and files needed for generating the API based on the OpenAPI contract provided by the server, including a subpackage generated by OpenApiGenerator;
► UI which includes graphical interface elements, the framework for dependency injection, and the blocks that bridge the graphical interface and application logic. Notably, the UI package is the only Flutter package in the application, while the others are Dart packages without graphical interface framework classes.

The Plugin Architecture

Plugin architecture is apattern that enables an application to be extended and customized through the addition of small, independent modules called plugins. This architecture promotes modularity and extensibility, allowing new features to be incorporated without altering the core system, thus enhancing flexibility and scalability. Plugins operate with well-defined interfaces, ensuring smooth interaction with the main application and other plugins, while dynamic loading capabilities allow these components to be added or removed at runtime.

Using DLL file loading at runtime, the server can load plugins that extend the system's functionality. These plugins can be developed independently and added to the system without recompiling the server. The server scans a specific folder for DLL files and loads them at runtime, making the system highly flexible and extensible. The plugins can be developed by third parties or by the system's owner, allowing for easy customization and adaptation.

Dynamic options available on the client injected by server plugin

This functionality is achieved by using the C# reflection mechanism, via the Assembly class. The server scans the plugins folder for DLL files and loads them using the Assembly.LoadFile method. The server then searches for classes that implement the IInjector interface and creates instances of these classes. The IInjector interface defines some methods that take the app's IServiceCollection as a parameter, allowing the plugins to register their services and dependencies. This mechanism ensures that the plugins can inject their services into the server's dependency injection container, making them available to the rest of the system. It's possible to create and inject any type of components (Mapper, Persisters, Repos, EventObservers, Controllers, etc.)

The modular architecture allows the creation of plugins that provide options displayed in the client application. When an option is selected, it triggers an action on the server. These options act as predefined endpoints created by the modules and callable from the client application. This setup facilitates defining and implementing modules that add specific functionalities, with each module offering one or more options shown in the user interface. Activating an option prompts the client to request the server to execute a specific action, enhancing the application's flexibility and extensibility.

Entities

Entites are structured following the Glovo API, with some modifications to better fit the system's requirements, ensuring a high level of compatibility with delivery apps that can be integrated with the system using (not yet implemented) plugins.

Entities are designed to be easy to understand and use by the restaurant's staff, ensuring a smooth transition to the new system. The menu is composed of:
► Sections - representing the menu sections, such as appetizers, main courses, desserts, etc. A product can be in multiple sections at the same time;
► Products - representing the dishes available in the restaurant, with information about the name, description, price and special flags;
► AttributeGroups - representing the groups of attributes that can be associated with a product, such as size, cooking preferences, etc. A product might have multiple attribute groups and a group might be associated with multiple products;
► Attributes - representing the attributes that can be associated with a product, such as small, medium, large, rare, medium rare, etc. An attribute can be associated with a single attribute group and a group might have multiple attributes;
► LocationGroups - representing the groups of locations within the restaurant such as the different rooms where customer serving might occur;
► Locations - representing the locations within the restaurant, such as the different tables where customers might be seated. A location can be associated with a single location group and a group might have multiple locations;

All menu related entities are being stored using temporal tables, ensuring that the system can provide historical data about the menu, allowing the querying of the menu state at any given time.

ER Diagram for database

An active order is composed by the list of ordered items, the location, the timestamp of the order. The orderedItem is composed by the product and, the list of selected attributes associated with the product and the flags for statuses (preparation status, delivery status, paid status).

A finished order is composed of similar elements as an active order, but with the addition of a flag signaling if the order was canceled. A finished order item does not have the flags for statuses, as the order is already finished.

The system is designed to be able to handle multiple orders at the same time, with the possibility of having orders with different menu versions associated with them.
